## Product direction — public league & team pages (plan tiers, target)

Most of this is **not built yet**; this section records **Basic / Pro / Enterprise** intent so public surfaces and billing stay aligned.

### Information architecture (target)

- **League home** — A **dedicated public home for the league**: the place for everything **browse, follow, and watch**—teams & rosters, standings/stats (tiered), featured games, **video/stream**, stream **overlays** (tiered), news/banner, and clear paths to **join** the season or **drop-ins**. **Target route:** e.g. `**/league/[slug]`** (exact path TBD). **Pro / Enterprise — custom fan domain (v1 shipped):** verified hostname on **`organizations.custom_domain`** serves league home at **`/`** via **`proxy.ts`** rewrite; **Phase 7** still carries **TLS automation**, **Clerk vanity host**, **Enterprise registrar assist**, and **SEO / canonical** polish.
- **Join** — `**/join/[slug]`** is for **people taking action**: **season registration**, **drop-ins**, and other signup flows. **Jersey number poll** (**Pro** or **Enterprise**) is **not** a `/join/...` page—players pick on the **public team page** while signed in (see **Phase 5**). It should read as **“sign up / participate,”** not the full league destination. League home and join **cross-link** (e.g. “Register” on league home → join flows; “League home” from join when helpful).
- **Shipped (Basic shell):** `**/league/[slug]`** — league home with **sticky tab nav** (Home, News, Schedule, **Stream**, Standings, Teams, About): Home = participate CTAs + league status; News = organizer updates; Schedule = combined season+drop-in list (+ venues/personalization); **Stream** — tab is **always visible** on **Basic**, **Pro**, and **Enterprise**. **Basic:** tab shows a **plan-locked** fan-hub preview (plain **Basic / Pro / Enterprise** wording; shared **`lib/public-plan-copy.ts`**) so visitors see what upgrades unlock—**no** embed + **no** live public box score on that tab. **Pro** or **Enterprise:** same tab is the **canonical fan hub** (optional **`StreamWithOverlay`** embed when a live game has a URL + **`LeaguePublicStreamFanBlock`** / **`PublicStreamBoxScore`** + **`?tab=stream&game=`** deep links; **`/league/[slug]/stream`** mirrors the tab and stays **locked on Basic** instead of redirecting away). **Standings** = **Basic** upgrade messaging vs **Pro** or **Enterprise** **Overview** (W/L + leaders) and **History** (final scores → **`?tab=stream&game={gameId}`** on this league); Teams = directory; About = evergreen CMS sections (history/media). News banner, season context, **team directory** → `**/league/[slug]/teams/[teamId]`**, links to `**/join/...`**. `**/join/[slug]`** root **redirects** to **`/league/[slug]`**; **season registration** and **drop-ins** use **`/join/[slug]/register`** and **`/join/[slug]/dropins`** respectively. **Jersey polls** (**Pro** or **Enterprise**): **`/league/[slug]/teams/[teamId]`** Overview + API (Phase 5). **Public team** tabs mirror the same idea: **Stream** stays in the row on **Basic** with a locked body; **Pro** or **Enterprise** get the embed path. Long-form CMS lives on league home only. Legacy `**/join/[slug]/teams/[id]**` redirects to `**/league/.../teams/[id]**`. Legacy **`/games/[gameId]/scoreboard`** redirects to the **Stream** tab with that **`game`** (no standalone scoreboard page).
- **Shipped (CMS v1):** Dashboard **League website** (`**/dashboard/league-site`**) and on-page editing at `**/league/[slug]?edit=1`** (owners / website editors) — **Save draft** / **Publish**; **hero** upload; **text**, **news**, **media** blocks with **Up / Down** reorder in dashboard and on-page editor. **Basic plan:** public league uses **MyLeaguePortal house branding** (**MyLeaguePortal Original** shell — cream + house green + light mode via **`getPublicThemeInputsForOrg`**); **no** custom hero / logo on public; **league-site save, publish, and uploads are API-blocked** (tier checks use shared **`lib/org-plan-tier.ts`** alongside settings, appearance, seasons, team logo, waiver PDF extract, and public **join** payloads); edit UI shows **locked** **Pro** previews and upgrade paths. **Pro** and **Enterprise (branding):** brand color, **six named presets** (**MyLeaguePortal Original** / **Classic / Bold / Soft / Bright / Midnight** — canonical id **`portal_original`** for MyLeaguePortal Original on `**organizations.league_theme_preset**`, migration `**20260506200000_league_theme_choice_ids.sql**`), **Bright / Midnight** via `**organizations.league_appearance_mode**` (migration `**20260506140000_league_appearance_mode.sql**`) synced with **Dashboard → Settings** and on-page **Save brand & look** (`**/api/league-org-appearance**`); optional **page fonts** via `**league_site_content**` (`**publicFontKey**`). **Theme tokens** are **contrast-harmonized** in **`lib/leagueTheme.ts`** (readable type on page + card surfaces; **Midnight** = dark shell + light foreground; **public hero** keeps near-white hero type on the dark gradient). **Preset pickers:** compact **pill rows** in Settings and on-page editor (descriptions on hover/`title`). **Public tab bar:** **centered wrap** (no horizontal scroll / scrollbar) and **no** full-width border under the row—only the **active** tab accent underline. Live **theme preview** while editing (owner) before save. `**GET /api/join/[slug]/hub`** includes `**leagueSite`**, `**plan`**, appearance fields. APIs: `**/api/league-site`** (GET returns `**appearance**` meta + `**maxGalleryImages**`; PUT 403 on Basic), `**/api/league-site/upload`** (403 Basic), `**/api/me/org-access**`, `**/api/organization-editors**`. Join surfaces use `**resolveThemePreset(brand, preset, appearanceMode)**` + `**getPublicThemeInputsForOrg**` for tier-aware theming. Dev seed: `**POST /api/dev/seed-teams-players**` — `**withLeagueSiteDemo: true**` upserts rich `**league_site_content**`. Migrations: `**20260505213000_league_site_content.sql**` (+ `**20260505213039**` no-op).
- **Future — Shop (not shipped):** Optional **Shop** on **league home** for **Enterprise** leagues first—see **Product direction — League shop & merchandising**. Basic/Pro do not include shop surfaces until product revisits tiering.
- **Future — Enterprise custom HTML block (not shipped):** Optional **“Custom HTML / embed”** section when adding blocks on **Home / News / About** — **Enterprise** only; **allowlisted sanitization** and API **`403`** below Enterprise. See **Enterprise → League website — custom HTML / embed block** in this doc.
- **Future — Enterprise HTML / custom theme & overlay (not shipped):** **Enterprise** may allow **sanitized HTML** or **sandboxed custom blocks** to extend **public league page theming** and **stream / score overlay** presentation beyond preset + token controls. See **Enterprise → Enterprise theme & stream overlay — HTML / custom blocks**.

### Basic

- **League home (Basic):** **House public theme** (**MyLeaguePortal Original** — cream shell, house green, light mode); **hosted-on** lockup instead of league logo on public; teams, registration, and drop-ins unchanged. **Custom league website** (hero, sections, fonts) unlocks on **Pro** or **Enterprise**; **Basic** visitors do not see org custom colors/hero on the public web.
- **Team directory** → **public team pages** at `**/league/[slug]/teams/[teamId]`** (identity, season, **roster** without contact info, **jersey poll** on **Overview** when open — **Pro** or **Enterprise**). **Roster on public** for all plans (names, jersey #, positions)—**no emails/phones on public** by default; plain roster (no stat strips). **Shipped:** tabbed layout (**Overview / Stream / News / Schedule / Roster / Stats** with `**?tab=`**); **Stream** uses the same **visible + locked on Basic** pattern as league home; **tab chrome** matches league home (**underline** vs **MyLeaguePortal Original** pill track — see Changelog **2026-05-21**). **Overview** can show **Watch live** when **`teams.stream_url`** (external) or a league **`live_game_id`** (opens **this team’s** **`?tab=stream`** on **Pro** or **Enterprise**; **Basic** still lands on locked hub copy until the league upgrades). **Last game** recap (Pro/Enterprise) lives on the **Stream** tab.
- **No public team/player stats** on Basic.
- **League schedule UI (single list):** Keep season + drop-in items in one list, but require strong **type indicators** (**League game** vs **Drop-in** / **Repeating drop-in**) and action-safe CTAs (e.g. reserve/join only appears on drop-ins).

### Pro

- **Live / final scores** on public league/team surfaces as a **teaser** (not necessarily full historical dumps everywhere—keep UX readable).
- **Five headline stats** on **Pro** public surfaces — **organizer-picked** (exactly **five** from **`MIN, PTS, 2PM, 3PM, FTM, AST, REB, STL, BLK, TOV, PF`**) stored in **`organizations.public_stream_primary_stat_keys`** (default **MIN / PTS / 3PM / TOV / PF**), edited under **Dashboard → Games** (Pro and Enterprise only). Those five always render **first (left)** for fans; remaining columns follow in a fixed order (Pro shows locked placeholders past the five). **Shipped on public team page** `**/league/[slug]/teams/[teamId]`**: Pro shows those **five** season totals + **W–L** + **league rank** + **last final** teaser; **Enterprise** shows **all** tracked columns in canonical order + **recent games** table.
- **Public / stream box score — stat columns (shipped):** On **Basic**, the league **Stream** tab (and team **Stream** tab) is **plan-locked**—fans do not see the live stat grid there until the org is on **Pro** or **Enterprise**. **`GET /api/public/games/[gameId]/box-score`** still exposes **`publicBoxScoreTier`**: **`basic`** = **roster-only** tables when that payload is used; **`pro`** = the **five** org-picked columns **first**, then **locked** headers/cells (🔒 / —) for the rest (**tap** → **Enterprise** upsell sheet); **`enterprise`** = **full** grid. **`publicStreamPrimaryStatKeys`** travels on that **`GET`**.
- **Optional headshot slots** on public roster cards—**organizer opts in** per league/team/player policy.
- **Team logo** upload on the public team page.
- **Brand-aware theme presets (Pro):** League page styling derives from league **brand color** in Settings. Generate **5 automatic theme presets** and let organizer choose one (no full free-form theme editing in Pro).
- **Preset behavior guardrail (Pro):** Presets must stay **brand-derived** (hue/tone variants from chosen brand color), not unrelated/random palettes.
- **Brand color change policy (Pro):** Brand color changes are **rate-limited monthly** (target default: **5/month**, configurable). Changing brand color regenerates the 5 presets for that league.
- **Optional logo-assisted branding (Pro):** Provide an **“Extract suggested brand color from logo”** helper; saved brand color remains the source of truth.
- **Video / stream:** featured **YouTube** and **Twitch** (embeds and/or prominent outbound links) on **league home** / team surfaces. **Shipped (MVP):** per-team **stream URL** on the public team **Stream** tab (`**teams.stream_url`**); **league home → Stream** tab is the **canonical fan hub** for **live box score + optional embed** on **Pro** or **Enterprise** (see Changelog **2026-05-14** and **2026-05-14 — Public Stream tab — plan-locked Basic**). **Basic** leagues keep the tab for discovery but see **locked** copy + **Dashboard → Settings** path. **Shipped:** when the league has a matching **`live_game_id`**, team **Overview → Watch live** opens **this team’s** **`?tab=stream`** (see **2026-05-21**). **Planned / polish:** further tie-ins when the org’s live game always matches the team fixture; **first-party streaming** remains **Enterprise** / Phase 7 scope.
- **iPhone / “true fullscreen” + score overlay (web constraint):** Safari does **not** allow a parent-page **custom HTML overlay** (live scores) on top of an embedded YouTube/Twitch player in **native video fullscreen**—fullscreen is owned inside the iframe. **Product stance:** keep **“Full screen with overlay”** as **immersive page mode** + optional **landscape lock** where browsers allow it; treat **player fullscreen** as the path for **system fullscreen** only (no site overlay). For **broadcast-style** “scores always on the video in true fullscreen,” organizers should **burn scores in OBS** (or similar) or use **first-party streaming** when that ships (**Enterprise** / Phase 7)—not something a web embed alone can fix on iOS.
- **Stream overlay:** **Basic overlay** for OBS/streaming—**template-based**, limited customization (on-brand, not full white-label).
- **Personalized schedule highlight:** **Shipped** on public league `Schedule` tab — signed-in players see `Your upcoming games` (priority strip) plus `You’re playing` markers on relevant season fixtures and reserved drop-ins while the full schedule stays visible to everyone.

### Planned mid tier (optional — billing not live)

- **Idea:** A **future optional plan** between **Pro** and **Enterprise** for leagues that want **more than Pro** (e.g. **full public box score stat columns**, other polish) **without** full Enterprise scope. **Price intent:** roughly **$59–69/mo** (TBD). Requires a real **`plan`** slug + Stripe Checkout + product gates before anything ships. **Until then,** shipped billing and UI use **Basic**, **Pro**, and **Enterprise** only (see **Plain-English glossary**).

### Enterprise

- **All stats** the platform can derive from recorded games, with **organizer toggles** (which stats appear where: **league home**, team page, **public Stream / box score**).
- **Sport-oriented analytics** (aggregates, trends) built from existing game data—scope grows with templates.
- **Auto season award maker**; **final weeks of season** prompts/notifications for organizers; **award race** callouts eligible for **league home** promotion.
- Public **team page**: **full edit** within supported templates/layout (copy, sections, branding)—not necessarily arbitrary HTML at first.
- **Theme controls (Enterprise):** Includes all Pro auto-brand behavior plus **fully customizable theme controls** (manual colors/tokens/section styling) with **unlimited edits**.
- **Enterprise theme & stream overlay — HTML / custom blocks (planned):** Optional **sanitized HTML** or **sandboxed custom blocks** to extend **public league page** theming and **stream / score overlay** layout & styling beyond the bullets above (**Enterprise** only; API + UI **`403`** below Enterprise). **Not shipped** — must align with **CSP / XSS** guardrails and **iOS / native player fullscreen** limits (see **Pro → iPhone / “true fullscreen” + score overlay**).
- **Video / stream:** **First-party / league-owned streaming** on the platform (full product—**scope TBD**). Complements Pro’s **YouTube/Twitch-first** positioning for leagues that want an owned destination.
- **Stream overlay:** **Fully customizable** overlays—layouts, branding, **sponsor placements**, advanced OBS/stream integration (see Phase 7).
- **League shop & merchandising:** **Enterprise** — public **Shop** for browse + optional on-site checkout when enabled (see **Product direction — League shop**).
- **AI-assisted setup & navigation:** **Enterprise** — plain-language copilot for schedule, news, teams, and rosters with confirm-before-apply (see **Product direction — AI-assisted setup**).
- **League website — custom HTML / embed block (Enterprise, planned):** Optional section type when adding blocks on **Home / News / About** for **sanitized** custom markup or **allowlisted** embeds (save + render pipeline must prevent **XSS**; size limits; Basic/Pro remain structured blocks only). **Not shipped** — product still to define **owner-only** vs **website editors**, exact tag/iframe policy, and API gates (**`403`** below Enterprise).
- Pairs with **Phase 7** custom domains and multi-admin for operational scale.

### Org “employees” / staff (not payroll)

- **Roadmap:** **Phase 7 — Multi-admin** — **Additional dashboard access** the **primary organizer grants and can revoke** (e.g. refs, scorekeepers, league staff). Delegated roles vs full org owner—not payroll “employees.”
- **Shipped (narrow):** **Website editors** — extra users who may edit **League website** only (`**organization_editors`**); not full multi-admin for seasons, teams, billing, etc.

### Messaging & in-app chat

- **Product stance (owner):** Real-time **group chat** (Messenger-style threads) is intentionally **late**—revisit when the business has **sustainable revenue** and you are ready to ship it as part of a **dedicated member/fan app** (native or high-quality **wrapped** web app), not as a rushed web-only sidebar.
- **Deferred / TBD:** Many leagues already use **Messenger / WhatsApp**; avoid a **group-chat clone v1** until **moderation**, **notification expectations**, and **mobile push** strategy are defined.
- **Placeholder intent (chat-adjacent):** **Enterprise** (or a **future optional mid-priced plan** between **Pro** and **Enterprise**, if we ship one) might later include **structured** league/team messaging (announcements, moderation, optional private team channel)—**evaluate at delivery time**, likely **after** opt-in **fan/member notification** rails exist (see below).

### Fan & member notifications (opt-in) — future

- **Why (separate from chat):** Fans and members should be able to choose **“tell me about…”** — upcoming games or drop-ins, **news**, **new season / registration**, **stats highlights**, and similar — **without** building Messenger first. This targets **useful pings**, not another inbox to babysit.
- **Delivery sequencing (typical):** **Transactional email** first (fits **Phase 6** “email first” direction) → **SMS** where budget and compliance allow → **web / PWA push** if installable web becomes a product bet → **reliable lock-screen push + sound** with a **future app** when chat and member identity warrant it.
- **Timing guardrail:** Ship **notification preferences + a small set of high-value triggers** (e.g. “your game tomorrow,” “registration opens”) **after** core signup, billing, and public league flows feel solid—natural home is **Phase 6** logistics work (alongside **Automated reminders**) **before** heavy **chat** scope. Per-trigger **Basic / Pro / Enterprise** gates stay **TBD**.
- **Cross-links:** **Team manager / club workspace → Calendar reminders to members**; **Phase 6 — Automated reminders**; this section stays **separate** from **Messaging & in-app chat** above.

### Team manager / club workspace (future — separate from league organizer)

**Persona:** **Team managers / captains / club leads** running **their team’s** logistics—not the **league-wide** organizer. This is **not** the same as **Phase 7 multi-admin** (league staff on the **organizer dashboard**).

**From manager interviews (prioritize when building):**

- **Calendar uploads / import** — Bring in `**.ics`** (and similar) so practice, league games, and team events live in one place; **subscribe / export** as a follow-on.
- **Calendar reminders to members** — Notifications tied to events (**email** first; **SMS / push** TBD—reuse Phase 6-style delivery when it exists). Aligns with **Product direction — Fan & member notifications (opt-in)** for broader opt-in alerts.
- **Automated polls + reminders** — e.g. **attendance / “who’s bringing X”** polls, scheduled **by default ~24 hours before game time** or at **offsets the manager chooses**; results and nudges go to **team members** as reminders (not only league-wide blasts).
- **Pre-set home game rules** — **Facility, arrival time, jersey colors, equipment checklists**, etc., configured under **team management** and shown on the **team workspace** (league-level defaults vs team overrides need a **single source of truth** rule—design before shipping).
- **Team front page** — A **home** for the team: **news** (team-only posts; optional surfacing of **league news** if the organizer allows), **next events**, quick links (polls, rules, files).

**Other ideas (validate with managers):**

- **Fundraising (solo team organizer workspace):** lightweight **team campaigns** (goal, story, share link) with optional **Stripe** donations or **catalog + outbound link** to an external store — evaluate legal/tax and whether this belongs in **Team manager** vs **league shop** (Enterprise) before scoping.
- **Next game** strip: opponent, time, **directions/map**, RSVP or poll shortcut.
- **League schedule (read-only)** on the team calendar with **team-only** overlays (practices, socials)—avoid duplicating editable league fixtures on the team side.
- **Lightweight file stash** (parking PDFs, permit scans, music links).
- **Simple roles** later (coach vs player vs parent) if one “member” view is too blunt.

**Status:** **Not started** — park after **League home**; **invites and permissions** (who is a team manager, roster membership) need explicit modeling so team tools don’t grant league-admin powers.

### League shop & merchandising (**Enterprise** — future)

**Status:** **Not started** — **Enterprise plan** for the shipped product; Basic/Pro do not expose shop until/unless tiering changes.

- **Goal:** A **Shop** area where members and fans **browse** league-curated items—uniforms / spirit wear, fundraiser products, **optional tickets or passes**, **donations**, or digital goods—surfaced on **public league** pages (and optionally **public team** pages if scoped).
- **Browse vs buy — TBD:** **v1** might be **catalog + outbound links** only (organizer-managed URLs to Shopify, Square, Spreadshirt, league vendor, etc.) so there is no inventory or tax on-platform yet. **Later option:** **native checkout** on MyLeaguePortal (Stripe Checkout / Payment Links, line items, receipts) once **fulfillment, sales tax, refunds, and payout routing** are defined—evaluate with legal and accounting before committing.
- **Operator UX:** Organizers (and delegated roles when Phase 7 expands) configure **which products or links appear**, copy/pricing display rules, and **visibility** (league-wide vs team-scoped). No silent charges—any purchase flow must be explicit and auditable.
- **IA:** Align with **Information architecture** bullet **Future — Shop** (tab or prominent section on `**/league/[slug]`**; exact route pattern TBD).

### AI-assisted setup & navigation (**Enterprise** — future)

**Status:** **Not started** — **Enterprise plan** for the shipped assistant; Basic/Pro continue with manual dashboard flows until/unless scope changes.

- **Goal:** An **AI setup / copilot** where organizers describe intent in **plain language** (e.g. “Add a bye before playoffs,” “Publish a rain-delay note to news,” “Split this roster into two balanced teams”) and the product returns **structured proposed actions** across **schedule**, **league news / CMS blocks**, **teams**, **players / roster**, and related entities—reducing time hunting through dashboard routes.
- **Safety:** **Human review + confirm** (or explicit **Apply** per entity) before any **mutating** or **public-facing** change; respect **RLS**, **plan limits**, and **roles** (no AI-only path to billing, payouts, org ownership, or cross-org data).
- **Scope creep guard:** Start with **read-heavy suggestions + draft content**; add **writes** one domain at a time (e.g. schedule drafts first, then news drafts, then roster moves with validation).
- **Relationship to Phase 6:** The **AI schedule generator** (natural language → season schedule proposal) is the **first technical wedge**; the **shipped product** for NL-driven scheduling and the broader assistant is **Enterprise**-gated alongside **League shop** on marketing and billing intent.

## Phase 1–3 — Foundation & core (**completed**)

### Infrastructure

- **Auth:** Clerk with a root **`proxy.ts`** request boundary (Next.js 16 **`clerkMiddleware`**; same matchers as before—public **`/join`**, **`/league`**, **`/api/join`**, **`/games`**, auth pages; protect **`/dashboard`** and **`/onboarding`**). The deprecated **`middleware.ts`** filename was renamed per [Proxy](https://nextjs.org/docs/app/api-reference/file-conventions/proxy). **Social sign-in:** **Google** and **Facebook** (Clerk Dashboard → **User & Authentication → Social connections**); app uses **`app/(auth)/sign-in/[[...rest]]/page.tsx`** (`<SignIn />`).
- **Data:** Supabase (PostgreSQL + RLS).
- **Billing:** Stripe subscriptions — **Checkout** (`**/api/stripe/checkout`**), **customer portal**, **webhooks** (`**/api/webhooks/stripe`**), **`organizations.plan`** + **`stripe_subscription_id`**. **Post-return sync:** **`POST /api/stripe/sync-checkout`** (Settings loads with **`?upgraded=true&session_id={CHECKOUT_SESSION_ID}`**) so **Pro/Enterprise** unlock immediately even if the webhook is slow. **Dev check:** **`scripts/verify-stripe-env.mjs`** (env presence + Price id lookup via Stripe API). **Production:** Dashboard webhook endpoint must exist for the **same mode** as **`STRIPE_SECRET_KEY`** so renewals, portal changes, and cancellations update the DB.

### Core management

- **Seasons, teams, players:** Full CRUD-style flows with **plan limits** enforced in UI/API.

### Season games & scoring

- **Live scoring:** Game clock, period (quarter) selector, per-player stats (**PTS, AST, REB, STL, BLK, TOV, PF**).
- **Shipped scoring operator UX:** fixed **5 starters per side** on the scoring page, jersey-first tap flow, and one-tap event buttons (**+3, +2, +1 FT**, plus AST/REB/STL/BLK/TOV/PF).
- **Shooting breakdown:** `player_game_stats` now tracks **2PM / 3PM / FTM** (`fg2m`, `fg3m`, `ftm`) with team scores kept in sync from player totals.
- **Shipped stream + overlay (fan-facing):** League **Stream** tab (**Pro** or **Enterprise**): **`StreamWithOverlay`** with **on-video** **`/games/[id]/overlay`** when a live game has a URL; **player stat tables** (**`PublicStreamBoxScore`**) below—duplicate score/Q/clock header omitted when the overlay is for the **same** live game (**`hideLiveGameHeader`**). **`GET /api/public/games/[gameId]/box-score`** + Realtime + poll align with **Dashboard → Games → scoring**. **`/games/[id]/overlay?embed=1`** remains the **OBS / broadcast** band. **Dashboard** public watch link (when org slug exists): **`/league/{slug}?tab=stream&game={gameId}`**. **Basic** uses the same nav + URLs but **plan-locked** tab/watch copy (no hidden tabs—see **Product direction**).
- **Lifecycle:** Scheduled → live → final.
- **Post-game:** Highlights UI and **Player of the Game** scoring formula.

### Drop-in essentials

- **Sessions:** Create/manage sessions; **recurring schedules** where implemented.
- **Day-of:** Check-in, payment tracking, **organizer-only standings** (includes **reputation points / tiers** — Gold / Silver / Bronze / warning — and inactive handling in dashboard).
- **Pro:** **Automated team builder** for balanced lineups.

### Public portal

- **Marketing:** Professional landing page (`/`).
- **Join (`/join/[slug]`):** Root **`/join/[slug]`** redirects to **`/league/[slug]`** (browse lands on league home). **Season registration** and **drop-ins** use **`/join/[slug]/register`** and **`/join/[slug]/dropins`** respectively. **Jersey poll** (**Pro** or **Enterprise**): public team **`/league/[slug]/teams/[teamId]`** Overview + **`POST /api/join/[slug]/teams/[teamId]/jersey-poll-preference`** (no separate join URL). Guests where configured for drop-ins; season signup one player per account; **no jersey # on season signup**.
- **League home:** `**/league/[slug]`** — **Basic** shell shipped (teams, rosters, news banner, participate links, **CMS v1** sections + optional hero background); **Pro** / **Enterprise** depth (stats, **unlocked Stream** hub, overlays) per **Product direction**. **`LeagueHomeContent`** keeps **React hook order** stable (all **`useMemo` / `useEffect`** before loading/not-found early returns—see Changelog).
- **Fans:** Public **box score** on **`/league/[slug]?tab=stream`** (and **`?game={gameId}`**) for **Pro** or **Enterprise**; **Basic** sees **locked** Stream + watch URL surfaces with the same **Settings** upgrade path. **`GET /api/public/games/[gameId]/box-score`** + realtime/polling aligned with **Dashboard → Games → scoring**. Legacy **`/games/[gameId]/scoreboard`** **redirects** to the league **Stream** tab for that game.

### Waivers

### Sport templates & per-sport statistics engine (**deferred — major initiative**)

**Product framing (same product, new stat spine):** **Organizations**, **seasons**, **dashboard shell**, **teams**, **players**, **drop-in sessions**, **join**, and **billing** stay the **same** mental model for organizers. What changes is **how a finished (or live) game records numbers**, **how the scorekeeper enters them**, and **every fan/broadcast read path** that aggregates or displays those numbers (public box score, stream tables, overlay, team season totals, Pro “pick five,” leaders, etc.). **Easier said than done** because today those paths all assume **one** row shape — swapping the spine still means touching **many** coordinated pieces, not one screen.

**Should we do this now?** Default: **defer** until signup / Stripe / standings / stream feel solid — the work is **large** (weeks+, not a polish pass) and touches almost every scoring-adjacent surface. **Pull forward** when **credibly recorded non-basketball game stats** is clearly the **#1** constraint on growth (not just label/copy).

**Organizer demand (capture — real waitlist):** Leagues waiting on **baseball** and **soccer** for more than **positions + marketing** — they need **stats that match how those games are scored**. **Swim club team managers:** often need **calendar, meets, reminders, and family comms** first; that path aligns with **Product direction — Team manager / club workspace** (Phase 5 / 6 adjacency) **unless** the ask is explicitly **per-meet swim splits / places** inside the **same** league game stat row — which is a **different** product depth; **validate interviews** before folding swim into this initiative alone.

**Why it is big:** Today **`player_game_stats`** + dashboard scoring + **`PATCH /api/games/[gameId]/stats`** + **`GET /api/public/games/[gameId]/box-score`** + **`PublicStreamBoxScore`** + **`/games/[gameId]/overlay`** + public team season aggregation (**`lib/public-team-page-payload.ts`**, **`lib/public-team-season-view.ts`**) + **`organizations.public_stream_primary_stat_keys`** (Pro “pick five”) all assume **one** basketball-shaped numeric row (**`pts`**, **`fg2m`**, **`fg3m`**, **`ftm`**, **`reb`**, **`ast`**, **`stl`**, **`blk`**, **`tov`**, **`pf`**, **`seconds_played`**). Soccer-only columns (e.g. SOG, cards), volleyball (kills, digs), football yardage, etc. require **new persistence**, **scorekeeper UI**, **API contracts**, **fan grids**, **overlay field layout**, **migrations + backfill rules**, and **marketing copy** that never promises a metric the button does not record.

**Phased delivery (recommended):**

1. **Spec & allowlists (no ship):** Per template, define **native** columns (stored + scored), **derived** team stats (from **`games`** only), and **explicitly out-of-scope** until a later phase — avoids conversion-killing “fake” labels.
2. **Schema + APIs:** Add template-specific columns **or** a versioned JSON stat blob keyed by **`sport_template_id`**; migrate existing leagues (**default basketball**).
3. **Dashboard scoring:** Replace / branch the scorer for non-basketball templates (mobile + desktop parity).
4. **Fan + broadcast surfaces:** **`PublicStreamBoxScore`**, **`GET /api/public/games/[gameId]/box-score`**, **`/games/[id]/overlay`**, public team **Stats**, **Leaders** strip semantics, **`public_stream_primary_stat_keys`** validation (allowed keys per template).
5. **Team / standings semantics:** Draws (**ties**), **PF/PA**, streaks, and “points vs goals” language — only after **`games`** + standings rules are explicit.

**Tier story (once built):** **Basic** stays preview-only; **Pro** = organizer-picked headline pack from **that template’s allowed keys**; **Enterprise** = full template grid + history — same upsell logic as today, grounded in **real** columns.

**Product note:** Optional **`flag_football`** (or **`football`** + variant metadata) should be decided **before** schema work so **`sport_template_id`** stays stable.

## Phase 9 — Platform admin / operations (**future — MyLeaguePortal staff only**)

**Not** the same as **Phase 7 multi-admin** (league staff invited by an organizer) or the **organizer dashboard**. This is an **internal** surface for **you / trusted operators** to see health and volume across **all leagues** on the platform.

**Status:** **Not started** — add after billing and core league flows are stable enough that aggregates are meaningful.

### Access & safety

- **Gate hard:** e.g. **Clerk allowlist** (specific user ids or org) + **server-only** APIs; no “secret URL” or client-side PIN. **Audit log** for sensitive actions (exports, any future impersonation).
- **Principle of least privilege:** read-mostly v1; destructive or PII-heavy actions behind explicit steps.

### Dashboard ideas (prioritize v1 vs later)

- **At a glance:** Count of **organizations (leagues)**; split by **`plan`** (**Basic / Pro / Enterprise**); **new orgs** in last 7 / 30 days; optional **Stripe** summary (MRR, active subs, failed payments) when webhooks are trustworthy.
- **People / activity (coarse):** Total **season players** or **profiles** if the schema supports it without heavy joins; **signed-in activity** proxies only if you add lightweight analytics—avoid guessing from Clerk alone.
- **Drop-ins & registration (aggregate):** Sessions created, signups, waitlist pressure — enough to spot “is anyone using this?”
- **Games / scoring:** Games recorded, live vs final — validates the scoring product is getting real use.
- **Reports / exports:** CSV or **scheduled** exports for accounting and support (orgs, plans, key dates) — **no** bulk PII export by default.
- **Suggestions & feedback:** Queue from an in-app **“Send feedback”** form or email capture — triage states (**new / in progress / done**), link to **org slug** for context.
- **Support tools (later):** **Org lookup** by slug or name; read-only **org detail** (plan, Stripe customer id, feature flags). **Impersonation** only if legally/contractually clear — defer until needed.

### Nice-to-have after v1

- **Error / health:** API error rates, cron last-run (`**/api/cron/**`), Supabase storage usage.
- **Churn risk:** Orgs on Pro with **failed renewal** or **canceled** Stripe status in the last N days.

### Plain-English glossary (for non-developers)

These terms show up in **changelog** lines and **agent** summaries. You do **not** need to memorize them—this table is the cheat sheet.

| Term | Simple meaning | Why it matters |
|------|----------------|----------------|
| **API** | The app’s **hidden web addresses** that load or save data (e.g. “save drop-in session,” “load league home”). Your browser calls them in the background. | When we say we “fixed an API,” we usually mean **data now saves correctly** or **only the right user can load that data**. |
| **API / security notes** | Short way of saying: **we checked who is allowed to see or change data** (e.g. only the **league owner** manages drop-ins; invited **website editors** cannot sneak into another league’s sessions). | Stops **accidental leaks** (wrong person sees another league’s private list) or **wrong account** changing the wrong league. |
| **CMS deploy checklist** | **CMS** = the **League website** editor (text, news, photos on the public league page). **Deploy** = pushing the live app + database to the internet (e.g. Vercel + Supabase). The **checklist** = “did the database tables exist? Can photos upload? Does Publish work?” | If the database or **photo storage bucket** isn’t set up on the **real** site, **Save / Publish** or **image upload** can fail even when it works on your laptop. |
| **`npm run build`** | **Packages the whole website** for production: compiles pages, checks types, fails if something is broken. | Run before a big release so **users never hit** a broken deploy. **Success** = the command ends with no errors (see **exit code** below). |
| **`exit_code: 0`** | After a terminal command finishes, **0** = **success** (everything OK). **Non-zero** (1, 2, …) = **something failed**—read the error text above it. | Same idea as “pass / fail.” You only need to know: **0 = good**. |
| **`eslint --max-warnings 0`** | **ESLint** = an automatic **code style and bug-sniffer** (unused variables, risky patterns). **`--max-warnings 0`** = **even small warnings count as failure** so the codebase stays strict and consistent. | Catches **sloppy mistakes** before they become user-facing bugs. **Pass** = clean enough for the project’s rules. |
| **Basic** | The **starter** league plan: public league home, teams, join/register, drop-ins, schedule, news, and house branding — **without** custom league-site design on the public web, **without** the full public **Stream** fan hub (video + live box score), and **without** deep public stats polish. The **Stream** tab still **appears** for fans; on **Basic** it is **locked** with plain-language upsell. | Fans and organizers see **“upgrade”** or **locked** hints where a feature needs **Pro** or **Enterprise**. |
| **Pro** | The **growth** plan: branded public experience, **unlocked** public **Stream** hub (embed + live stats), headline stats you pick, jersey polls, custom domain (v1), and more — see **Product direction** above. | Billing and feature gates use the word **Pro** the same way as **Dashboard → Settings**. |
| **Enterprise** | The **full** plan: everything in **Pro** plus the widest stats surfaces, registrar-style options, and long-term Enterprise-only ideas (shop, AI, custom HTML) where the roadmap says so. | Same as **Pro** for many fan-facing stream features — copy often says **“Pro or Enterprise”** when both unlock the same thing. |
| **`lib/public-plan-copy.ts`** | One **TypeScript** file that holds the **exact sentences** for plan-locked **Stream** upsells (league tab, **`/league/[slug]/stream`**, team tab) so **Basic / Pro / Enterprise** wording stays consistent. | When visitor copy changes, update **this file** (and this **ROADMAP**) instead of editing three separate UI files by hand. |
| **Fan / member alerts** | Optional **emails** (and later **SMS** or **phone notifications**) when something you signed up for changes—**games**, **news**, **season signup**, **stats highlights**, etc. **Not** the same as **chat**. | Roadmap: **Product direction — Fan & member notifications**; first build is likely **Phase 6** with **email-first** delivery, **before** Messenger-style chat or a member **app**. |

Use this when validating `**league_site_content`**, organization_editors, and `**league-site`** storage are live.
